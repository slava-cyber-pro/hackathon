# BudgetSphere — Implementation Guide

## Project Overview

BudgetSphere is a collaborative financial tracking PWA. See `PROJECT_OVERVIEW.md` for features and `ARCHITECTURE.md` for technical design.

## Code Principles

- **Small functions**: every function does one thing, max ~20 lines. If it's longer — extract.
- **One responsibility per file**: models, schemas, routes, services each get their own file per domain.
- **No business logic in routes**: routes validate input, call a service, return a response. That's it.
- **No HTTP concerns in services**: services take plain args, return data or raise domain exceptions.
- **Type everything**: full type hints on all Python functions. Strict TypeScript on frontend (`noUncheckedIndexedAccess`).
- **No dead code**: don't leave commented-out code, unused imports, or placeholder functions.
- **Fail explicitly**: raise specific exceptions, never swallow errors silently.

## Backend Rules

### Project Setup
- Python 3.12+, managed with `pyproject.toml` (no `requirements.txt`).
- Use `ruff` for linting + formatting (replaces black, isort, flake8).
- All async — async def for routes, async SQLAlchemy sessions, async Redis.

### FastAPI
- App created via factory function in `app/main.py` using lifespan context manager.
- All routes under `app/api/v1/`, each domain in its own file.
- Use `APIRouter` with prefix and tags in each route module.
- Use dependency injection (`Depends`) for DB sessions, current user, etc.
- Route functions should be 5-10 lines max: parse input → call service → return schema.
- Use `status_code` parameter on route decorators, not magic numbers in responses.
- Global exception handlers in `app/utils/exceptions.py` for domain errors.

### Models (SQLAlchemy)
- One model per file in `app/models/`.
- Use `mapped_column()` with explicit types, not legacy `Column()`.
- UUIDs as primary keys (`uuid7` for time-ordered IDs).
- Use `Mapped[type]` annotations on all columns.
- All relationships defined with `relationship()` and `back_populates`.
- Enums defined as Python `enum.Enum` and mapped to PostgreSQL `ENUM` types.
- `__tablename__` always set explicitly.
- Add `__repr__` for debugging.
- Export all models from `app/models/__init__.py` so Alembic picks them up.

### Schemas (Pydantic)
- One schema file per domain in `app/schemas/`.
- Separate `Create`, `Update`, `Response` schemas (e.g., `TransactionCreate`, `TransactionUpdate`, `TransactionResponse`).
- Use `model_config = ConfigDict(from_attributes=True)` on response schemas.
- Validate at the schema level: amounts > 0, dates not in future where applicable, string lengths.
- Use `Annotated` types for common patterns (e.g., `PositiveDecimal = Annotated[Decimal, Field(gt=0)]`).

### Services
- One service file per domain in `app/services/`.
- Services are plain async functions (not classes) grouped by domain file.
- Take `AsyncSession` as first parameter, then domain-specific args.
- Return model instances or plain data — never HTTP responses.
- Raise domain-specific exceptions (defined in `app/utils/exceptions.py`), not `HTTPException`.
- Keep each function focused: `create_transaction()`, `get_transactions()`, `update_transaction()`, `delete_transaction()`.

### Dependencies (`app/api/deps.py`)
- `get_db()` — yields an async session, handles commit/rollback.
- `get_current_user()` — extracts and validates JWT from `Authorization: Bearer` header, checks `token_issued_after`.
- `get_current_active_user()` — wraps `get_current_user()` + checks active status.
- All return typed objects, not dicts.

### Auth Implementation
- Stateless JWT only — **never** store tokens in Redis or DB.
- Access token: 15 min TTL, HS256, payload: `{"sub": user_id, "iat": timestamp}`.
- Refresh token: 7 days TTL, rotated on every refresh call.
- Verify by signature + expiry + `token_issued_after` check. No DB/Redis lookup for validation.
- Passwords: `passlib[bcrypt]` with `CryptContext`.
- Login returns both access and refresh tokens.
- `POST /auth/refresh` accepts refresh token, returns new token pair.

### Database & Migrations
- Async engine via `create_async_engine` with connection pool settings.
- `async_sessionmaker` with `expire_on_commit=False`.
- Alembic configured for async with `run_async` in `env.py`.
- **Always** generate migrations via `alembic revision --autogenerate -m "description"`.
- Review autogenerated migrations before applying — Alembic can miss some changes.
- Seed default categories in a migration or startup hook, not in application code.

### Redis
- Used for: rate limiting, WebSocket pub/sub, background task queues.
- **Not used for**: token storage, session caching, analytics caching.
- Use `redis.asyncio` client.
- All Redis operations wrapped in try/except — app must work if Redis is temporarily down (graceful degradation).

### Observability
- `prometheus-fastapi-instrumentator` added in `main.py` lifespan.
- Custom metrics registered for business events (transaction created, user registered, etc.).
- `/metrics` endpoint exposed for Prometheus scraping.
- Structured logging with `structlog` — JSON format in production, pretty-print in dev.

### Error Handling
- Define domain exceptions in `app/utils/exceptions.py` (e.g., `NotFoundError`, `PermissionDeniedError`, `BudgetExceededError`).
- Map domain exceptions to HTTP status codes in global exception handlers.
- Never expose internal errors to clients — return generic messages with correlation IDs.
- Always log the full exception server-side.

## Frontend Rules

### Project Setup
- Vite + React 18 + TypeScript in strict mode.
- `pnpm` as package manager.
- ESLint + Prettier for linting/formatting.
- Path aliases: `@/` maps to `src/`.

### Component Guidelines
- Functional components only with TypeScript props interfaces.
- One component per file, file name matches component name.
- Max ~50 lines per component. If larger, extract sub-components.
- Props interfaces named `{ComponentName}Props` and exported.
- Use `React.memo` only when profiling shows a real perf issue — not preemptively.

### State Management
- **Server state**: TanStack Query for all API data. Define query keys in a central `queryKeys.ts`.
- **Client state**: Zustand for UI state (sidebar open, filters, modals). One store per domain.
- **No prop drilling** beyond 2 levels — use context or Zustand instead.
- Mutations via `useMutation` with `onSuccess` invalidation of related queries.

### API Layer
- Base client in `src/api/client.ts` with interceptors for auth token injection and refresh logic.
- Per-domain API modules (`src/api/transactions.ts`, etc.) — each exports typed functions.
- All API functions return typed responses matching backend schemas.
- Handle 401 globally in the client interceptor — trigger token refresh or redirect to login.

### Routing
- React Router v6 with lazy-loaded route components.
- Protected routes via an `AuthGuard` wrapper component.
- Route constants defined centrally — no hardcoded paths in components.

### Forms
- React Hook Form for all forms.
- Zod schemas for client-side validation, matching backend Pydantic constraints.
- Show inline validation errors, not alerts.

### Charts
- Recharts for all visualizations.
- Each chart type in its own component under `src/components/charts/`.
- Charts accept data as props — no data fetching inside chart components.
- Use `ResponsiveContainer` wrapper on all charts for mobile compatibility.

### Styling
- Tailwind CSS utility-first. No custom CSS files unless absolutely necessary.
- shadcn/ui for primitives (Button, Input, Dialog, etc.) — customized via Tailwind, not CSS overrides.
- Mobile-first: default styles for mobile, use `sm:`, `md:`, `lg:`, `xl:` for larger breakpoints.
- Dark mode support via Tailwind `dark:` variant + class strategy.
- Consistent spacing: use Tailwind spacing scale, don't use arbitrary values.

### PWA
- `vite-plugin-pwa` for service worker generation and manifest.
- Icons: at minimum 192x192 and 512x512 PNG.
- `display: "standalone"` in manifest for native app feel.
- Register service worker in `main.tsx`.

## Docker

- `docker-compose.yml` for development (hot-reload for both frontend and backend).
- `docker-compose.prod.yml` for production (Nginx for frontend, gunicorn + uvicorn workers for backend).
- Backend `Dockerfile`: multi-stage build, non-root user, health check.
- Frontend `Dockerfile`: multi-stage build (node build → nginx serve), non-root user.
- `.dockerignore` files in both `backend/` and `frontend/` — exclude `node_modules`, `__pycache__`, `.venv`, tests.
- Environment variables via `.env` file (not committed) with `.env.example` as template.
- Health checks on all services.
- Named volumes for PostgreSQL and Redis data persistence.

## Testing

### Backend Testing
- **Framework**: `pytest` with `pytest-asyncio`, `httpx` for async API tests.
- **Test database**: separate PostgreSQL database, created/torn down per test session. Never mock the database.
- **Fixtures** in `conftest.py`: `db_session`, `client` (httpx AsyncClient), `authenticated_client`, `sample_user`, `sample_team`.
- **Structure**: mirror `app/` structure — `tests/test_auth.py`, `tests/test_transactions.py`, etc.
- **What to test**:
  - Every service function: happy path + edge cases + error cases.
  - Every API endpoint: correct status codes, response shapes, auth enforcement, validation errors.
  - Auth: token generation, validation, refresh rotation, expired token rejection, `token_issued_after` enforcement.
  - Permissions: team role enforcement, user can only access own data (or team data if member).
  - Budget limits: warning thresholds, exceeded limits.
- **Coverage**: minimum 80% enforced via `pytest-cov`. Run with `pytest --cov=app --cov-report=term-missing`.
- **Naming**: `test_{action}_{scenario}` (e.g., `test_create_transaction_valid`, `test_create_transaction_negative_amount_rejected`).

### Frontend Testing
- **Framework**: Vitest + React Testing Library.
- **What to test**:
  - Components: renders correctly, user interactions trigger expected behavior, conditional rendering.
  - Hooks: custom hooks return correct data and handle loading/error states.
  - Forms: validation messages appear, submission calls correct API function.
  - API client: interceptors handle 401 refresh and token injection.
- **Mocking**: mock API layer (`msw` for network mocking), never mock React internals.
- **E2E**: Playwright for critical flows — register, login, create transaction, view charts, team invite.
- **Coverage**: Vitest coverage with `@vitest/coverage-v8`, minimum 80%.

### Test Commands
```bash
# Backend
cd backend && pytest --cov=app --cov-report=term-missing -v

# Frontend
cd frontend && pnpm test --coverage

# E2E
cd frontend && pnpm exec playwright test

# Full stack (Docker)
docker compose -f docker-compose.test.yml up --abort-on-container-exit
```

## UI Design (Stitch MCP)

Stitch is connected as an MCP tool for generating and managing UI designs before implementation.

### Workflow
1. **Create project**: `mcp__stitch__create_project` — one Stitch project for BudgetSphere.
2. **Create design system**: `mcp__stitch__create_design_system` — define colors, fonts, roundness, color mode to match our Tailwind/shadcn theme.
3. **Generate screens**: `mcp__stitch__generate_screen_from_text` — generate each page design from a text prompt describing the screen's purpose, layout, and components.
4. **Edit screens**: `mcp__stitch__edit_screens` — refine generated screens with follow-up prompts.
5. **Generate variants**: `mcp__stitch__generate_variants` — explore layout/color/typography alternatives before committing to a design.
6. **Apply design system**: `mcp__stitch__apply_design_system` — ensure all screens use consistent tokens after design system updates.

### Design System Config
- **Primary color**: finance-appropriate (blue/teal palette).
- **Fonts**: Inter (headlines + body) — matches our Tailwind config.
- **Roundness**: `ROUND_EIGHT` — modern, not too rounded.
- **Color mode**: `LIGHT` as default (dark mode variant to follow).
- **Color variant**: `TONAL_SPOT` — balanced, professional feel.

### Screens to Design
Design each screen for both **DESKTOP** and **MOBILE** device types:

| Screen          | Key Elements                                                      |
| --------------- | ----------------------------------------------------------------- |
| Login           | Email/password form, register link, app branding                  |
| Register        | Name/email/password form, login link                              |
| Dashboard       | Summary cards (balance, income, spending), mini charts, recent transactions |
| Transactions    | Filterable table/list, add button, category chips, date range picker |
| Investments     | Portfolio allocation chart, investment list, add form              |
| Budgets         | Category budget cards with progress bars, add/edit limits          |
| Team            | Member list, invite form, role badges, team spending overview     |
| Settings        | Profile edit, password change, notification preferences           |
| Analytics       | Full-page charts — spending breakdown, income trend, balance over time |

### Design-to-Code Rules
- Stitch designs are reference — implement with shadcn/ui components + Tailwind, not pixel-copy.
- Match Stitch's color tokens to Tailwind theme config (`tailwind.config.ts`).
- Use Stitch screen designs to inform component structure and layout hierarchy.
- Generate mobile + desktop variants to validate responsive breakpoints before coding.

## Git Workflow

- Branch naming: `feature/short-description`, `fix/short-description`.
- Commits: conventional commits (`feat:`, `fix:`, `refactor:`, `test:`, `docs:`, `chore:`).
- PR into `main` — squash merge.
- CI runs lint + type check + tests on every PR.

## File Naming Conventions

- **Backend**: `snake_case.py` for everything.
- **Frontend**: `PascalCase.tsx` for components, `camelCase.ts` for utilities/hooks/stores, `camelCase.ts` for API modules.
- **Tests**: `test_{module}.py` (backend), `{ComponentName}.test.tsx` (frontend).
